# 実例：Qorum は、実際に何をするのか

英語版：[../cases.md](../cases.md)

README は仕組みを抽象的に説明しています。このページは具体版——ゲートが実際に「何のため」にあり、このリポジトリ自身のゲートを自分に通したとき何が起きたか。

## 何のためか（問題の形）

AIと速く進めている。エージェントが変更を出し、別のエージェントがレビューし、あなたが merge する。たいていは問題ない——誤字直し、文言調整、小さなリファクタ。ゲートの役目は、そこを遅くしないこと。

問題は、見た目は同じなのに「大丈夫じゃない」少数です。

- 秘密情報を読めてしまう CI 設定の一行変更
- APIキーの読み込み方を「ちょっとだけ」変える修正
- 新しいものを静かに引き込む依存関係の更新
- レビュー役がざっと見ただけの、千行のリファクタ

速いAIレビューには、これも全部「OK」の緑チェックに見える。Qorum の答えは「もっと厳しくレビューしろ」ではなく——**リスクで仕分け、安全なものは流し、危ないもの（CI・秘密情報・依存・マイグレーション・不可逆）だけは止めて人間を待つ**。レビュー役のAIは、そこを通せない。

これが全体です。**安全な大半はAIの速度で流れ、危険なものは素通りできない。**

## このリポジトリで実際に起きたこと（ゲートを、自分自身に）

このリポジトリは、説明している仕組みそのもので作られています——一つのAIが実装し、別のAIが制約下でレビューし、人間が承認する。だから、自分自身に向けた。公開の3ラウンド。

**ラウンド1——ゲートに盲点があった。** レビュー役のAIがゲート自身のルールを点検し、どの段にも当てはまらないケースを見つけた。ちょうど4行を削除する変更が、どの段にも一致しない。「低」は削除ゼロ、「中」は3まで、「高」は5以上でしか発火しない。4は分類されないまま——分類が仕事のはずのツールに空いた穴。作者は気づいていなかった。「高」が中の上限を超えるものを拾うよう修正。→ commit [`c873605`](https://github.com/shiyamaz/qorum-method/commit/c873605)

**ラウンド2——docs は正しく、config が違った。** README は「大きな差分や依存ファイルは高リスク」と書いていた。だが実際の `.qorum.yml`——人がコピーするファイル——には、行数の閾値といくつかの依存マニフェストが抜けていた。config をコピーした人は、docs の約束より弱い保護しか得られない状態だった。実際に使うファイルにルールを入れて修正。→ commit [`c540b95`](https://github.com/shiyamaz/qorum-method/commit/c540b95)

**ラウンド3——そして、人間が止めた。** レビューはなお続き、今度はあらゆる断定をヘッジに弱めにきた。正しい指摘もあった（でっち上げの「80% / 20%」という数字を削った）。一方で、opinionated な文書から声を削りにきた部分もあった。人間（承認役）は、誠実な修正は採り、残りは退けた。その判断——**AIレビューに従う価値が、どこで尽きるか**——こそ、ゲートが人間に残すものです。→ commit [`6b4804a`](https://github.com/shiyamaz/qorum-method/commit/6b4804a)

ゲートが完璧だ、という話ではありません。盲点が、公開の場で、見つかり・名指しされ・直された——そして最後の一言は人間に残った、という話です。

## 正直な但し書き

上のコードのゲートは、実際に回して、直した部分です。同じ3段階を、コード以外の判断——戦略・採用・価格——に向けるのは、同じ"形"ですが、同じようには検証されていません（[beyond-code.md](beyond-code.md)）。構造を使い、自分の仕事で何が後戻りできるかから、自分の階層を引いてください。
