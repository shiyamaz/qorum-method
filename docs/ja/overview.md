# Qorum Method ｜ 日本語概要

Qorum Method は、AIに実装・レビュー・壁打ちを任せる時代に、「AIが進めてよいこと」と「人間が必ず決めること」をリスク別に分けるための、軽量な意思決定ガバナンスです。

これは英語 [README](../../README.md) の直訳ではなく、日本語で書き直した概要です。思想を少し前に出していますが、最後は実装の証拠に戻ります。

---

## これは何か

ひとことで言うと、**「AIに何を任せるか」ではなく「AIに何を決めさせないか」を先に決める方法**です。

案を作り、レビューし、最後に通すかを決める。この役割を3つに分けます。実装役とレビュー役はAIでも自分でもよく、人間が必ず引き受けるのは、後戻りできない判断（高リスク）の承認です。

- **実装役（Implementer）**：案を作る。AIでも自分でもいい
- **レビュー役（Reviewer）**：実装役とは別の主体。別のAIでも、自分でもいい。改善案を出すのではなく、粗を探す
- **承認役（Approver）**：人間。後戻りできない判断だけを引き受ける

そして、変更や判断を**リスクで3段階に分ける**。低リスクはAI同士で流し、高リスクは人間に戻す。これだけです。

一つだけ動かせない原則があります。**レビュー役は低・中リスクを通してよい。でも高リスクを通せるのは、人間の承認役だけ。**

## なぜ必要か

ひとりで動いていると、レビューしてくれる相手がいません。「自分のPRは自分で見ればいい」は、誰もが持っていると言い張り、夜11時には誰も守れない規律です。

かといって、AIに全部承認させるのは、レビューがない状態より危険です。**何でも通すレビュー役は、レビューの"実体"なしに"安心"だけを作る**からです。CIの設定も、秘密鍵の扱いも、千行のリファクタも、AIは自信たっぷりに「OK」と言います。

Qorum はその第三の答えです。**変更ごと・判断ごとに、どこまで任せていいかをリスクで決め、危ないものだけは確実に人間を通す。** 夜11時の自分が言い訳して飛ばせない一線を、先に引いておく。

## 具体的に、何を止めるのか

ふだんの変更——誤字直し、文言の調整、小さな修正——は、AI同士で速く流していい。ゲートの役目は、そこを遅くしないことです。

問題は、見た目は同じなのに、止めるべき少数です。

- 秘密情報を読めてしまう CI 設定の一行変更
- APIキーの読み込み方を「ちょっとだけ」変える修正
- 新しいものを引き込む依存関係の更新
- レビュー役がざっと見ただけの、千行のリファクタ

速いAIレビューには、これも全部「OK」の緑チェックに見えます。Qorum の答えは「もっと厳しくレビューしろ」ではなく、**リスクで仕分けて、危ないもの（CI・秘密情報・依存・マイグレーション・不可逆なもの）だけは人間で止める**こと。レビュー役のAIは、そこを通せない構造にしておきます。

これが全体です。**安全な大半はAIの速度で流れ、危険な数件は素通りできない。**

（このゲートを実際に自分自身へ通した記録は [`docs/cases.md`](../cases.md)（英語）にあります。）

## どう使うか（開発の場合）

3段階の目安です（設定例は [`examples/qorum.yml`](../../examples/qorum.yml)）。

- 🟢 **低リスク**：少数ファイル・削除なし・保護対象に触れない → レビュー後、実装役がそのままマージ（前提条件を満たせば）
- 🟡 **中リスク**：通常の実装変更 → レビュー後、通知して一拍置いてからマージ
- 🔴 **高リスク**：大きな差分・保護対象（CI／秘密情報／依存／マイグレーション／インフラ）・不可逆 → **人間の承認が必須。AIはマージ禁止**

迷ったら、下げない。低か中で迷えば中、中か高で迷えば高。安全側に倒します。

レビュー役のAIには、設計させない。「差分の妥当性／ルール違反／リスク階層／退行リスク／不整合」の5点だけを、短く見させる。**レビュー役は検査官であって、設計者ではありません。**

## 開発の外へ

実は、これは開発の話ではありません。開発（PR・レビュー・マージ）は、**一番わかりやすい適用先**にすぎない。すでにパイプラインがあり、差分という具体物があり、マージという行為があるから、説明しやすいだけです。

同じ3段階は、AIと進めるどんな意思決定にも写せます。

- 🟢 **低**：文言案・論点整理・比較表・調査メモ → AI同士で進めていい
- 🟡 **中**：施策・優先順位・顧客向け資料・採用要件 → AIレビューの後、人間が軽く見る／一晩置く
- 🔴 **高**：事業方針・価格・人事・法務・資金繰り・ブランドの不可逆な判断 → AIは助言まで。決裁は人間

AI活用の多くは「もっと任せよう（delegate more）」に寄っています。Qorum が立っているのは逆側、「**任せる範囲を縛る（bound the delegation）**」です。本当に要るのは、AIに"決めさせない"範囲を、自分の言葉で引くことでした。

詳しい写し方は [`docs/beyond-code.md`](../beyond-code.md)（英語）。

## 証拠と野心を分ける

正直に線を引きます。**開発のゲートは、実際に何度も回して鍛えたもの**です。この repo 自身、AIレビューに自分の分類不能ケース（どの基準にも当てはまらない判断）を掘り当てられ、直しています。

一方、意思決定への一般化は、同じ"形"を広い的に向けたもの——構造は確かですが、全領域で検証済みの台本ではありません。だから上の表をそのまま輸入しないでください。**あなたの仕事で何が後戻りできるか**から、自分の階層を引く。

表の中身ではなく、問いの方が本体です。

> あなたは今、AIに、どこまで「決めさせて」いますか。ただ、自信たっぷりに答えたから、というだけの理由で。
